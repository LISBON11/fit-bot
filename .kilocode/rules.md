1. **Context7 First**: Всегда при использовании технологии используй Context7 mcp.
2. **Уточнения**: Если данных недостаточно — задавай уточняющие вопросы.
3. **Инициатива**: Если есть более элегантное решение — предложи альтернативу и сравни.
4. **Архитектура**: После планирования архитектуры дождись подтверждения перед реализацией.
5. **Стек по умолчанию**: Используй Node.js и TypeScript.
6. **Линтинг**:
   - Используй ESLint. Все написанное должно проходить проверки `eslint`, `tsc`, `prettier`. Не используй `/* eslint-disable` в коде, типизируй по всем правилам.
   - Не добавляй off правил в eslint для any
7. **Логирование**: Обязательно добавляй логи в релевантных местах (используем `pino`), используй адекватный уровень логирования.
8. **Тестирование**:
   - Каждая функция обязательно должна быть покрыта unit тестами, использовать `jest` + `mock`.
   - Юниты складывай в папку `__tests__` на уровне с покрываемой утилитой.
   - После правок актуализируй тесты и документацию (если пирамида тестирования настроена).
   - После каждой задачи запускай линтеры и автотесты (если конфигурация присутствует).
9. **Документирование**:
   - Всегда добавляй TSDoc к коду с умеренной детализацией (на русском языке).
   - Storybook стори к компонентам (в будущем, когда будет любой frontend).
   - Архитектурные решения и изменения **всегда** протоколируй в `docs/architecture/decisions`.
   - При изменении бизнес решений **всегда** протоколируй их в `docs/business/decisions/`.
10. **Глобальные скиллы**: При поиске скиллов для реализации задачи не забудь проверить глобальные скилы в `~/.agents/skills/`.
11. **Тесты**: Не "костялый" моки через eslint-disable-next-line или @typescript-eslint/no-explicit-any или ts-expect-error и прочие. Мок должен полностью совпадать по структуре с обьектом, который он заменяет. Доставай моки из библиотеки `jest-mock-extended` или из самих библиотек, если они их поставляют для тестов. 12.**Сигнатуры функций**:
    Всегда используйте паттерн 'Object Parameters' (передача аргументов через один объект), если у функции или метода изначально предполагается более 1 аргумента.

- Исключения: Функции стандартных библиотек, обработки событий `(req, res, next)`, встроенных методов вроде `reduce((acc, curr) => ...)` или коллбэков массивов.
- Пример: Вместо `function createUser(name: string, age: number, role: string)` пишите `function createUser(params: { name: string; age: number; role: string })`.

13. Всегда используй CLI‑прокси rtk вместо «сырых» команд, когда это возможно.

- Все команды git, ls, grep/rg, cat/read, просмотр логов и запуск тестов по умолчанию вызывать через rtk.
- Если команда не поддерживается rtk или что‑то ломается, можно один раз повторить её без rtk как fallback.
  - Примеры правильных команд
    - rtk git status
    - rtk git diff / rtk git log
    - rtk ls
    - rtk grep <pattern> etc.

14. **Общее**:

- Пиши артефакты вроде walkthrough, implementation plan или task list на русском.
- Не коммить по завершению задачи.
