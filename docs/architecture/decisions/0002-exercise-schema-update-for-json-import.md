# Обновление схемы базы данных для Упражнений

## Контекст (Context)

Пользователь захотел загрузить упражнения из файла `filtered_exircise_ru.json` в базу данных. В JSON-файле содержались данные, которых не было в текущей схеме `Exercise` Prisma (инструкции, уровень сложности, тип упражнения), а также отличался тип поля целевых мышц (`primaryMuscle`).

## Решение (Decision)

1. Было решено расширить модель `Exercise` в схеме Prisma:
   - Обновить поле `primaryMuscle` (поиск одной мышцы) на массив строк `primaryMuscles String[]`.
   - Маппить поле `force` из JSON напрямую в существующее поле `movementPattern` БД.
   - Добавить новые поля:
     - `level String?` (например, beginner/intermediate)
     - `instructions String[] @default([])` (инструкции по выполнению)
     - `exerciseType String? @map("exercise_type")` (тип тренировки, в JSON это `category`)
2. В скрипт загрузки начальных данных `prisma/seed.ts` добавлена логика парсинга JSON, где поле `name` (уже переведенное на русский) используется в качестве `displayNameRu`, а ключом `canonicalName` выступает транслитерированная версия этого названия в виде slug.
3. Поскольку скрипт использует custom Database-адаптер (`getPrismaClient` внутри `src/config/database.ts`), мы импортировали приложение напрямую в скрипт сидирования вместо базового создания `new PrismaClient()`.

## Последствия (Consequences)

- Существующие значения `primaryMuscle` в таблице `exercises` были удалены при переходе на массив `primary_muscles` (об этом свидетельствует предупреждение миграции). Это было приемлемо, так как данные все равно пересобираются.
- Теперь база данных корректно отражает все полезные свойства из новой выгрузки упражнений, что позволит боту/клиенту предоставлять более детальную справку по ним.
- Скрипт `seed.ts` больше не содержит хардкодных данных упражнений и обращается к внешнему JSON-файлу.
