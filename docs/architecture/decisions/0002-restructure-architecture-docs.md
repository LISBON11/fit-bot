# 2. Реструктуризация архитектурной документации

Дата: 2026-02-23

## Статус

Принято

## Контекст

Использование папки `docs/adr/` одновременно с глобальным файлом `SYSTEM_DESIGN.md` в корне проекта создавало стилистическое противоречие: актуальный дизайн системы и записи о принятых решениях находились в несопоставимых местах дерева исходников, что затрудняло навигацию. Кроме того, создание папки с названием `adr` (Architecture Decision Records) и сохранение в ней любых других архитектурных доков (или попытка объединить текущий дизайн и историю) противоречило семантике самого термина ADR.

## Решение

Была создана родительская директория `docs/architecture/`, внутри которой архитектурная документация была разделена по принципу жизненного цикла:

1. `docs/architecture/decisions/` — для исторических записей архитектурных решений (ADR). Эти документы фиксируют контекст, неизменяемы и описывают причины выбора решений.
2. `docs/architecture/design/` — для актуального дизайна системы (`SYSTEM_DESIGN.md` и сопутствующие файлы). Этот раздел постоянно обновляется вместе с эволюцией системы.

## Последствия

- Более понятная структура документации для разработчиков и AI-агентов.
- Четкое логическое разделение на History/Log (`decisions`) и Snapshot/Current State (`design`).
- Потребовалось обновление всех ссылок в ROADMAP.md и README.
