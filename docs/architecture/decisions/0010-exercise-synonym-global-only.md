# 10. Удаление пользовательских синонимов из ExerciseSynonym

Дата: 2026-03-06

## Статус

Принято

## Контекст

Модель `ExerciseSynonym` содержала поля `isGlobal` и `userId`, предполагая возможность хранения как глобальных (для NLU), так и пользовательских синонимов. Однако:

- Реализация создания пользовательских синонимов так и не была написана.
- Передача смешанного списка синонимов в контекст LLM создаёт риск «отравления» — персональные ошибочные названия одного пользователя попадают в контекст другого.
- Пользовательские маппинги уже хранятся в `UserExerciseMapping` — это де-факто и есть персональные «синонимы».

## Решение

- Удалены поля `isGlobal` и `userId` из таблицы `exercise_synonyms` (миграция `remove_user_specific_synonyms`).
- Таблица `ExerciseSynonym` теперь хранит **только глобальные** синонимы, которые передаются в NLU-контекст.
- Для пользовательских маппингов используется таблица `UserExerciseMapping`.

## Последствия

### Положительные

- Чистота NLU-контекста: в промпт попадают только проверенные глобальные алиасы.
- Упрощена логика `findSynonyms` и `resolveExercise` — нет необходимости фильтровать по `userId`/`isGlobal`.
- Схема данных точнее отражает реальное использование.

### Отрицательные

- Нет механизма «личных» синонимов (пока). Если в будущем потребуется — нужна отдельная модель или возврат поля.

## Будущее

Периодически анализировать `UserExerciseMapping` по частоте использования. Наиболее популярные маппинги рассматривать как кандидаты для добавления в глобальные синонимы. Задокументировано в `docs/HOTELKI.md`.
