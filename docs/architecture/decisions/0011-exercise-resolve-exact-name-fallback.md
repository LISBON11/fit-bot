# 11. Детерминированный fallback по имени упражнения в resolveExercise

Дата: 2026-03-06

## Статус

Принято

## Контекст

NLU (DeepSeek) может корректно распознать название упражнения из речи (например, «жим лёжа»), но из-за нестабильности LLM не проставить поле `mappedExerciseId` в JSON-ответе. В таком случае `originalName` содержит правильное название, но код его не разрешает и показывает пользователю кнопки выбора — хотя ответ известен.

Ранее алгоритм `resolveExercise` после поиска по `ExerciseSynonym` сразу переходил к fuzzy-поиску (`searchSimilar`), минуя прямое сравнение с полями `Exercise.canonicalName` / `Exercise.displayNameRu`.

## Решение

Добавлен шаг 3 в алгоритм `resolveExercise`:

```
1. UserExerciseMapping (точный маппинг пользователя)
2. ExerciseSynonym (глобальная таблица синонимов)
3. findByExactName: точный поиск по canonicalName / displayNameRu (регистронезависимо) ← NEW
4. searchSimilar (fuzzy) → ambiguous с кнопками
5. not_found
```

Новый метод `ExerciseRepository.findByExactName` выполняет `prisma.exercise.findMany` с `OR` по обоим полям и режимом `insensitive`.

## Последствия

### Положительные

- Снижается количество ненужных disambiguation-кнопок при галлюцинациях NLU.
- Шаг полностью детерминированный — нет риска ложных срабатываний как у fuzzy-поиска.
- Минимальная нагрузка (один простой SELECT по индексированному полю).

### Отрицательные

- Добавляет один дополнительный запрос к БД (только когда синонимы не найдены, т.е. редко).
