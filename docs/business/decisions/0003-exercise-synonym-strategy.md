# 0003. Стратегия синонимов упражнений

Дата: 2026-03-06

## Текущий статус

Принято

## Контекст

Пользователи называют упражнения по-разному: «жим», «жим лёжа», «bench», «жим штанги лежа» и т.д. NLU (DeepSeek) получает список глобальных алиасов и должна распознавать упражнение по `mappedExerciseId`. Однако LLM-модели нестабильны и могут:

- Правильно извлечь название (`originalName`), но не проставить `mappedExerciseId`.
- Вернуть неизвестное название, близкое к реальному («жим лех» вместо «жим лёжа»).

Ранее всё разрешение шло через таблицу синонимов и fuzzy-поиск. Неочевидная принадлежность `ExerciseSynonym` (глобальный vs. пользовательский) создавала архитектурный шум.

## Решение

Принята трёхуровневая модель хранения и разрешения упражнений:

| Уровень | Хранилище                                  | Назначение                                       |
| ------- | ------------------------------------------ | ------------------------------------------------ |
| 1       | `UserExerciseMapping`                      | Личные подтверждённые маппинги пользователя      |
| 2       | `ExerciseSynonym`                          | Глобальные алиасы для NLU-контекста              |
| 3       | `Exercise.canonicalName` / `displayNameRu` | Детерминированный fallback при галлюцинациях NLU |
| 4       | Full-text / fuzzy search                   | Последний шанс перед disambiguation              |

`ExerciseSynonym` содержит **только глобальные** синонимы. Пользовательские маппинги живут в `UserExerciseMapping` и не передаются в NLU-промпт.

## Дальнейший план

Периодически (раз в месяц / по достижению порога использования) анализировать `UserExerciseMapping`:

- Выбирать маппинги с `useCount >= N` (порог определяется по реальным данным).
- Наиболее популярные `inputText` добавлять как глобальный синоним в `ExerciseSynonym`.
- Это позволит органически обогащать словарь NLU на основе реального поведения пользователей.

Задача зафиксирована в `docs/HOTELKI.md`.
